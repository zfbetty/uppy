Uppy 1.5: Facebook and OneDrive (beta) support, upload cancellation

Friday Oct 11, 2019

In the past two months, we have halved our open issue count and worked on a much more robust approach to upload cancellation. Members of the community also contributed a bunch of new localizations: Czech, Danish, Greek, Indonesian, and Swedish!

We are also releasing beta versions of new remote providers for Facebook and OneDrive. Please try them out!

Issue busting

Our issue tracker was getting pretty full over the past year, topping out at about 120 open issues. Some had been open for months without any reply! Issues were being opened at such a rate that it was almost too much for our small team to handle. So, we sat down to go through everything together and started setting up a process to make sure our issue list doesn‚Äôt get out of hand again in the future.

When opening a new issue, we now ask you to mark it as a bug or a feature request. The issue will also get a ‚ÄúTriage‚Äù label, indicating that an Uppy team member needs to sort it. If there are many ‚ÄúTriage‚Äù issues open at any point, we will discuss them in our weekly team call. New issues will also be assigned to a specific team member as soon as possible, so each of us has a much smaller and more manageable list of issues to keep track of.

To keep the issue tracker focused on Uppy features and bugs, detailed support questions should go to Transloadit‚Äôs community forum.

Duplicate files

Up until now, if you added a file that had been already added to Uppy some time before ‚Äî i.e., a file with the same name/type/modifiedDate etc. ‚Äî the old file would be silently replaced with a new one. This changes with v1.5:

  1. Uppy now throws a warning if a file with the same file.id already exist. The new file will not be added and the old file will not be replaced.
  2. We are adding the file.meta.relativePath to file.id. relativePath exists on files that have been dropped together with a folder. This means that if you drop a folder with folder/a.jpg and folder/subfolder/a.jpg, both a.jpgs will be added, since they come from different folders and this kind of duplication might be intended.

See PR #1767 for details.

Cancellation

Our old approach to limiting the amount of simultaneous uploads had some problems. Uploads were queued by chaining Promises, but they did not respect cancellation. So, if you cancelled an upload while some files were still waiting to start, those files would actually start uploading even though they had already been removed. Obviously, that‚Äôs pretty bad!

PR #1736 addresses this by ripping out the old limiting code and replacing it with a new, cancellation-aware upload queue. We‚Äôve also added a browser test to make sure that this thing stays resilient:

Lerna lerna lerna

In August, we bit the bullet and used Lerna‚Äôs lerna link convert feature. Now, all the packages in our monorepo depend on each other, using file:../packagename/ paths instead of version numbers. The entire repository is one big dependency tree in npm‚Äôs view. Npm can install the entire thing on its own, without lerna bootstrap‚Äôs help! Since npm manages the entire tree, the package-lock.json file in the repository root contains everything we need. We can now use the faster npm ci install command on CI, saving more than sixty seconds on every run :tada:

As part of this effort, we also added all our examples to the lerna-managed tree, so you no longer have to do npm install in example folders to use them.

Revoke Companion‚Äôs Provider access on logout

Up until now, when users clicked ‚Äúlogout‚Äù after using a remote Provider, such as Google Drive or Dropbox, Uppy would destroy the access token. But on the Provider‚Äôs side, Uppy was still allowed access. Should the user click ‚ÄúConnect to Google Drive‚Äù again, no extra confirmation was required.

After a discussion in #1563, we are expanding the logout process. Now, when users log out of remote Providers through Uppy, Companion will destroy the token as before, but also attempt to revoke the access from the corresponding Provider. In the event that the Provider doesn‚Äôt support access revoke via it‚Äôs API, Uppy will display a helpful message informing users how they can revoke the permissions manually.

See #1843 for details.
